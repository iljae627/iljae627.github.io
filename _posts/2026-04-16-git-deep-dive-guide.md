Git은 단순히 소스코드를 저장하는 저장소를 넘어, 파일 시스템의 **스냅샷(Snapshot)**을 관리하는 정교한 그래프 구조체입니다. **Learn Git Branching**의 로컬 및 원격 레벨을 모두 클리어하며 체득한 Git의 핵심 명령어들을 내부 동작 원리와 함께 심도 있게 정리합니다.

---

## 1. 커밋 트리의 기초 (The Core Concepts)

### **git commit**
- **내부 동작**: 현재 인덱스(스테이징 영역)의 파일 상태를 트리 객체로 변환하여 저장하고, 이를 가리키는 커밋 객체를 생성합니다.
- **델타(Delta) 저장**: Git은 저장 공간을 효율적으로 관리하기 위해 변경된 부분(델타)만 기록하며, 각 커밋은 부모 커밋의 해시값을 참조하여 체인을 형성합니다.

### **git branch & checkout**
- **branch (참조)**: 브랜치는 특정 커밋 해시를 가리키는 **40글자의 포인터 파일**일 뿐입니다. `git branch -f <name> <target>`을 통해 특정 위치로 강제 재설정(Force Move)이 가능합니다.
- **checkout (HEAD 이동)**: `HEAD`는 현재 체크아웃된 커밋 혹은 브랜치를 가리키는 포인터입니다. 브랜치명이 아닌 커밋 해시를 직접 체크아웃할 때 발생하는 **Detached HEAD** 상태를 주의해야 합니다.

---

## 2. 히스토리 재구성 및 병합 (Branch Manipulation)

### **git merge vs git rebase**
- **Merge (머지)**: 두 브랜치의 공통 조상(Base)과 각 브랜치의 끝단을 합쳐 **두 개의 부모**를 가진 새로운 커밋을 만듭니다. 병합된 기록이 선명하게 남습니다.
- **Rebase (리베이스)**: 현재 브랜치의 커밋들을 대상 브랜치 위로 하나씩 **복사하여 재배치**합니다. 머지 커밋 없이 히스토리를 한 줄로 깔끔하게 정리할 수 있지만, 이미 공유된 커밋에 대해서는 사용을 지양해야 합니다.

### **git cherry-pick <commit>...**
- **선별적 복사**: 전체 브랜치를 병합하지 않고, 특정 해시값을 가진 커밋만 골라 현재 `HEAD` 아래에 복사본을 생성합니다. 버그 수정 사항 등만 급하게 반영해야 할 때 가장 효율적입니다.

---

## 3. 원격 저장소와의 동기화 (Remote Mechanisms)

### **git fetch vs git pull**
- **Fetch (페치)**: 원격 저장소의 최신 커밋들을 다운로드하여 `origin/main`과 같은 원격 추적 브랜치만 갱신합니다. **로컬 파일은 변경되지 않습니다.**
- **Pull (풀)**: `fetch` 이후 곧바로 현재 브랜치와 `merge`를 수행합니다. 원격의 내용을 즉시 로컬에 반영하고 싶을 때 사용합니다.

### **git push (푸시)**
- **내용**: 내 로컬 저장소의 커밋들을 원격 서버에 업로드하고 원격 브랜치 포인터를 갱신합니다. 원격 저장소에 내 로컬에 없는 새로운 커밋이 있다면 거절(Reject)되므로, 반드시 `pull`을 먼저 수행하여 병합해야 합니다.
